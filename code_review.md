## 三、详细问题清单

### 🔴 高风险问题（1个）

#### 问题1：前端硬编码 API Key

| 项目 | 内容 |
|------|------|
| 问题等级 | 高风险 - 安全 |
| 涉及文件 | `utils/ai.js` |
| 问题描述 | 前端代码中直接写入阿里云 DashScope API Key，任何人都可以通过浏览器控制台或查看源码获取 |
| 风险影响 | API Key 泄露后，他人可盗用账号调用AI接口，产生高额费用 |
| 修复状态 | ✅ 已修复 |

**修复方案**：
- 生产环境下前端不持有任何 API Key
- 所有 AI 调用通过云函数中转
- API Key 仅存储在云函数的 uni-config-center 中

---

### 🟡 中风险问题（4个）

#### 问题2：输入验证不充分，XSS 防护不足

| 项目 | 内容 |
|------|------|
| 问题等级 | 中风险 - 安全 |
| 涉及文件 | 所有云函数的 `sanitizeInput` 函数 |
| 问题描述 | 原 `sanitizeInput` 只过滤了 `<>` 两个字符，对 XSS 攻击防护不足 |
| 风险影响 | 恶意用户可能通过构造特殊字符注入脚本 |
| 修复状态 | ✅ 已修复 |

**修复方案**：
- 增加过滤字符：`& " ' \` ;`
- 增加关键词过滤：移除 `script`
- 增加事件处理器过滤：移除 `onclick=`、`onload=` 等

---

#### 问题3：缺少编程语言白名单验证

| 项目 | 内容 |
|------|------|
| 问题等级 | 中风险 - 安全/质量 |
| 涉及文件 | `generate`、`review`、`explain` 云函数 |
| 问题描述 | 用户传入的 language 参数没有验证，可能传入任意字符串 |
| 风险影响 | 异常语言名称可能导致 AI 输出不可预期，也可能被用于注入攻击 |
| 修复状态 | ✅ 已修复 |

**修复方案**：
- 定义 8 种支持语言的白名单：JavaScript、TypeScript、Python、Java、Go、Rust、C++、C#
- 自动规范化输入的大小写（如 `javascript` → `JavaScript`）
- 支持常见缩写：js → JavaScript、ts → TypeScript、cpp → C++

---

#### 问题4：代码重复严重，工具函数分散

| 项目 | 内容 |
|------|------|
| 问题等级 | 中风险 - 可维护性 |
| 涉及文件 | 10个云函数 |
| 问题描述 | 每个云函数都重复定义了 `sanitizeInput`、`createLogger` 等相同的工具函数，约50行重复代码 |
| 风险影响 | 修改一个函数需要同时改10个文件，容易遗漏出错，维护成本高 |
| 修复状态 | ✅ 已修复 |

**修复方案**：
- 创建 `common/utils/index.js` 公共工具模块
- 统一提供：createLogger、sanitizeInput、hashPassword、generateTokenWithExpiry、validateLanguage、getUserByToken 等函数

---

#### 问题5：缺少单元测试

| 项目 | 内容 |
|------|------|
| 问题等级 | 中风险 - 质量保障 |
| 涉及文件 | 全局 |
| 问题描述 | 项目没有任何单元测试，修改代码后无法验证是否引入新bug |
| 风险影响 | 重构或修复bug时容易引入回归问题 |
| 修复状态 | ✅ 已修复 |

**修复方案**：
- 使用 Jest 测试框架
- 编写 24 个测试用例，覆盖 5 个核心工具函数
- 测试覆盖率：sanitizeInput (8个用例)、hashPassword (4个)、generateTokenWithExpiry (3个)、validateLanguage (4个)、getUserFriendlyError (5个)

---

### 🟢 低风险问题（4个）

#### 问题6：错误处理简单，缺少结构化日志

| 项目 | 内容 |
|------|------|
| 问题等级 | 低风险 - 可维护性 |
| 涉及文件 | 所有云函数 |
| 问题描述 | 使用简单的 `console.log` 输出错误，格式不统一，难以排查问题 |
| 风险影响 | 线上出问题时日志信息不足，定位困难 |
| 修复状态 | ✅ 已修复 |

**修复方案**：
- 创建 `createLogger` 工厂函数
- 统一日志格式：`[函数名] INFO/ERROR/WARN: 消息`
- ERROR 级别自动输出错误堆栈

---

#### 问题7：缺少配置中心管理

| 项目 | 内容 |
|------|------|
| 问题等级 | 低风险 - 工程化 |
| 涉及文件 | 各云函数顶部配置 |
| 问题描述 | API Key、模型名称、接口地址等配置分散在各个云函数中 |
| 风险影响 | 修改配置时需要逐个云函数修改，容易遗漏 |
| 修复状态 | ✅ 已修复 |

**修复方案**：
- 创建 `uni-config-center/ai-config/index.js` 统一配置中心
- 集中管理：API Key、模型名称、接口地址、默认参数等

---

#### 问题8：CI/CD 流水线缺少测试步骤

| 项目 | 内容 |
|------|------|
| 问题等级 | 低风险 - 工程化 |
| 涉及文件 | `.github/workflows/ci.yml` |
| 问题描述 | CI/CD 只做语法检查，没有运行单元测试 |
| 风险影响 | 有问题的代码可能被合并到主分支 |
| 修复状态 | ✅ 已修复 |

**修复方案**：
- CI 流程增加：npm install → ESLint 检查 → Jest 单元测试 → 覆盖率报告上传
- 使用 Node.js 20.x 环境

---

#### 问题9：图片生成 API 响应格式解析错误

| 项目 | 内容 |
|------|------|
| 问题等级 | 低风险 - Bug |
| 涉及文件 | `utils/ai.js` 的 `generateImage` 函数 |
| 问题描述 | 阿里云万相 API 实际响应格式为 `output.results[0].url`，代码错误地解析为 `output.urls` |
| 风险影响 | 图片生成功能完全不可用，一直显示"图片生成失败" |
| 修复状态 | ✅ 已修复 |

**修复方案**：
- 增加响应日志打印，便于排查问题
- 兼容多种响应格式：`output.results[0].url`、`output.urls`、`output` 字符串
