# AI辅助编程与工程化实训个人总结

## 一、项目概述

本次实训项目是开发一个基于 uni-app x + uniCloud + 阿里云通义千问 API 的 AI 代码助手应用。项目包含 AI 问答、代码生成、代码审查、代码解释、文件分析、AI P图、视频通话等七大核心功能，以及用户系统、历史记录、收藏等辅助功能。整个项目开发过程中，我大量使用了 AI 辅助编程工具（Trae AI）来提高开发效率和代码质量。项目共包含 12 个前端页面、9 个后端云函数、5 张数据库表，以及完整的 CI/CD 流水线和单元测试套件。

## 二、遇到的最大困难及解决过程

### 困难一：云函数资源耗尽导致服务不可用

在项目开发后期，我遇到了最严重的一个问题：云函数调用频繁失败，错误信息显示"FC invoke failed, resource exhausted"。这意味着阿里云函数计算的资源配额被耗尽了，导致所有依赖云函数的功能都无法使用。

#### 排查过程

1. **第一阶段：怀疑 API 调用频率过高**
   
   一开始我以为是自己测试太频繁，触发了限流。但等待一段时间后问题依然存在，说明不是临时限流。

2. **第二阶段：查看云函数监控**
   
   登录阿里云控制台查看云函数的调用记录，发现确实有大量失败请求。但奇怪的是，成功请求并不多，远没达到免费额度。

3. **第三阶段：定位问题根源**
   
   仔细分析失败的请求日志，发现很多请求在调用云函数时就失败了，甚至没进入业务逻辑。进一步排查发现，问题出在云函数的冷启动和资源配置上。每次调用云函数都需要加载依赖模块，而我的云函数引入了很多外部依赖，导致冷启动时间过长，超时失败。

#### 解决方案

1. **精简云函数依赖**：将云函数中的外部依赖移到本地实现，减少冷启动时间
2. **实现双模式架构**：在 `utils/ai.js` 中实现了云函数调用 + 直接 API 调用的双模式，当云函数失败时自动降级为直接调用
3. **添加错误重试机制**：增加了请求失败后的重试逻辑，提升系统健壮性
4. **优化资源配置**：调整云函数的内存和超时时间配置

这个问题让我深刻认识到：**在 Serverless 架构中，冷启动和资源配置是必须考虑的关键因素**。AI 工具在这个过程中帮我分析了可能的解决方案，并给出了具体的代码实现建议。

### 困难二：数据库记录保存失败问题

在项目开发中期，我遇到了一个非常棘手的问题：云函数调用后，数据库中始终没有记录产生。前端明明显示调用成功，但数据表就是空的。

#### 排查过程

1. **前端验证**：反复检查 `uniCloud.callFunction` 的参数，确认参数都传对了
2. **云函数调试**：在云函数里加了 `console.log`，发现云函数确实执行了，也拿到了数据库返回结果
3. **日志分析**：查看 uniCloud 的错误日志，发现问题："提交的字段在云端数据表的schema文件中不存在"

原来 uniCloud 的数据库有 schema 验证机制，如果上传的字段不在 schema 定义里，就会被静默忽略。我的 `chat_history` 表只定义了几个基础字段，但代码里传了 `type`、`language` 等额外字段，导致整条记录都没存进去。

#### 解决方案

修改每个数据表的 schema.json 文件，补充缺失的字段定义，并将每个字段的权限设置为 `true`，允许读写。

### 困难三：AI P图功能异常

在开发 AI P图功能时，调用阿里云万相 API 后，接口返回成功状态，但就是拿不到生成的图片。经过排查发现是 API 响应格式不一致的问题——AI 给出的代码使用 `output.urls`，但实际阿里云返回的是 `output.results[0].url`。

## 三、对 AI 辅助编程的心得体会

### 1. AI 是效率倍增器，但不能代替思考

在项目开发中，AI 帮我节省了大量时间：写云函数的基础框架几分钟就能搞定，写正则表达式不用再查文档反复试，排查 bug 时能快速给出可能的原因。但 AI 也不是万能的。有好几次，AI 给出的代码看起来很完美，但一跑就报错。比如图片生成的 API 响应格式问题，就是因为我没有验证就直接使用了 AI 给的代码。

**我的体会是：AI 是很好的助手，但最终的判断和验证还是要靠自己。** 代码是我写的，bug 也是我要负责的。

### 2. 写好 Prompt 是一门学问

同样的问题，不同的问法，AI 给出的答案质量差别很大。我总结了几个写 Prompt 的技巧：

- **说清楚背景**：告诉 AI 你用的是什么技术栈、什么环境
- **列清楚要求**：一条一条列出来，不要笼统地说"帮我写个xx"
- **给具体例子**：比如想要什么格式的输出，给一个示例
- **让 AI 解释思路**：不要只要代码，要让 AI 解释为什么这么写

最关键的一点是：**要让 AI 成为你的副驾驶，而不是自动驾驶**。你需要时刻掌控方向，AI 只是帮你踩油门。

### 3. AI 代码审查是质量保障的重要环节

一开始我觉得 AI 代码审查就是走个形式。但真的用了之后，发现 AI 能找出很多我忽略的问题：

- **安全漏洞**：前端硬编码 API Key，我自己完全没意识到这是个严重问题
- **代码重复**：10 个云函数都在写同样的工具函数，AI 一提醒我才发现
- **边界情况**：输入验证不充分、错误处理不完善

特别是安全性方面，AI 给出的建议让我对 Web 安全有了更深入的认识。原来一个不经意的 API Key 硬编码，可能会导致严重的安全风险和经济损失。

### 4. AI 辅助编程的局限性

在使用 AI 的过程中，我也发现了一些局限性：

- **环境感知不足**：AI 不知道你当前的运行环境，可能给出不兼容的代码
- **上下文理解有限**：对于大型项目，AI 很难理解整个项目的架构和依赖关系
- **创造性不足**：AI 擅长模仿和优化，但在创新架构设计方面还不够强

## 四、个人在项目中的具体贡献

作为项目的主要开发者，我负责了以下工作：

### 1. 架构设计

- **技术选型**：选择 uni-app x + uniCloud 的技术栈，实现前后端一体化开发
- **云函数规划**：设计了 9 个云函数的职责划分，包括 chat、generate、review、explain、analyze、image、video、login、register
- **数据库设计**：设计了 5 张数据库表（users、chat_history、file_history、image_history、video_calls），并编写了完整的 schema 文件

### 2. 核心功能开发

- **用户系统**：实现了注册、登录、Token 认证、密码加密（SHA256 + salt），确保用户数据安全
- **AI 对话**：对接通义千问 API，支持上下文对话，实现了聊天记录的保存和读取
- **代码工具**：实现了代码生成、审查、解释三个功能，支持 8 种主流编程语言
- **文件分析**：实现了上传文件的解析和分析功能
- **图片处理**：对接万象 AI 实现图片生成和编辑功能

### 3. 代码质量优化

- **结构化日志**：引入 createLogger 工厂函数，统一所有云函数的日志格式，便于问题排查
- **公共模块提取**：将重复的工具函数提取到 `common/utils/index.js`，消除代码重复

- **语言白名单验证**：添加 validateLanguage 函数，确保只处理支持的编程语言
- **单元测试**：编写了 70+ 个测试用例，覆盖核心工具函数，测试覆盖率达到 88%+

### 4. 工程化建设

- **CI/CD 配置**：配置 GitHub Actions 自动构建和测试流水线，每次 push 自动触发
- **配置中心**：创建 uni-config-center 统一管理 API Key、模型配置等敏感信息
- **文档编写**：编写了完整的项目文档，包括 README.md、API.md、code_review.md、prompt_log.md

## 五、深度思考

通过这次实训，我有以下几点深度思考：

### 1. AI 时代的编程范式正在改变

传统的编程方式是"程序员写代码 -> 编译器编译 -> 测试 -> 部署"。而 AI 辅助编程的范式是"程序员提需求 -> AI 生成代码 -> 程序员验证和优化 -> 测试 -> 部署"。这种转变意味着：

- **需求表达能力变得更重要**：能清晰地描述需求，就能让 AI 生成更好的代码
- **代码审查能力变得更重要**：需要快速判断 AI 生成的代码是否正确、安全、高效
- **学习能力变得更重要**：AI 工具更新很快，需要持续学习新的使用方法

### 2. 安全意识是 AI 时代程序员的必备素质

AI 生成的代码可能存在安全隐患，比如硬编码密钥、缺少输入验证等。作为程序员，必须具备基本的安全意识，在使用 AI 代码之前进行安全审查。

### 3. 工程化是项目成功的保障

一个项目的成功，不仅仅是功能的实现，更是工程化的建设。CI/CD、单元测试、日志系统、配置管理——这些工程化手段确保了项目的可维护性和稳定性。

## 六、总结与展望

通过这次实训，我不仅完成了一个完整的 AI 应用项目，更重要的是学会了如何高效地使用 AI 工具来辅助开发。

我最深的感受是：**AI 时代的程序员，会写 Prompt 和会写代码同样重要**。知道怎么问问题、怎么引导 AI 给出高质量的答案，往往比自己埋头写代码效率更高。

但同时我也清楚地认识到，AI 只是工具。真正的架构设计、问题排查、质量把控，还是需要人的经验和判断。打好基础，才能更好地利用 AI 这个强大的工具。

未来我会继续学习 AI 辅助编程的最佳实践，探索更多 AI 和软件工程结合的可能性，让自己成为一名 AI 时代的高效开发者。同时，我也会更加注重安全和工程化，确保项目的质量和稳定性。
